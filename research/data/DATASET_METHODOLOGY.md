# Dataset Methodology

## Two-Part Evaluation Dataset

### Part 1: SWE-bench Verified (standardized)
- Source: princeton-nlp/SWE-bench_Verified (HuggingFace)
- 489 tasks from 9 repos, chronologically ordered per-repo
- Ground truth patches available for objective evaluation
- Used for: single-task baseline comparison against Devin/Codex/OpenHands
- Eval set: 25 tasks (5 per repo × top 5 repos)

### Part 2: Synthesized Longitudinal Chains (novel)
- Purpose: evaluate cross-task knowledge accumulation (memory + retrospectives)
- Each chain = 10 sequential tasks in the same repo where later tasks benefit from earlier task knowledge

#### Generation Method (v1 — current)
Chain tasks generated by `claude -p` with repo name only. Fast but shallow — tasks may not reflect actual codebase structure.

#### Generation Method (v2 — TODO, more rigorous)
1. Clone the target repo locally
2. Run `claude -p` WITH access to the repo (`--add-dir .`) so it reads actual code
3. Prompt: "Read this codebase. Generate 10 realistic enhancement/bugfix tasks that form a longitudinal chain. Each task should be concrete (specific files, functions, tests) and later tasks should benefit from knowledge gained in earlier ones."
4. This produces repo-specific, grounded tasks — not generic template tasks

#### Why v2 matters for the paper
- Reviewer objection: "your synthesized tasks don't reflect real codebase complexity"
- v2 response: "tasks are generated by an LLM that has read the actual codebase, referencing real files, functions, and test patterns"
- This is analogous to how SWE-bench uses real GitHub issues — we use LLM-grounded synthetic issues

### Total Dataset
- SWE-bench: 25 eval tasks (standardized comparison)
- Longitudinal chains: 30 tasks (3 chains × 10 tasks, cross-task memory eval)
- Total: 55 tasks per condition × 7 conditions = 385 experimental runs
